# 配置文档-多端同步

本文件从操作者视角描述 Test 2：一个 Team Memory server，多 个 Hermes client。不要手工编辑 JSON。server-local 配置必须通过 setup 命令交互式写入并激活。

相关测试：[[多端同步手动测试.md#多端同步手动测试]]。

## 测试环境一键维护

重新构建当前工作树并替换 `hermes-local`、Hermes A/B 和共享 Team Memory service：

```powershell
npm.cmd run hermes:test:redeploy
```

清空 Test 1 local 与 Test 2 shared server 的业务记忆，但保留所有 `rbac_*` 核心身份、管理员、角色、授权和 session 数据：

```powershell
npm.cmd run hermes:test:clear-memory
```

完整的删除范围、`-NoCache` 用法和安全边界见 [[配置文档-local.md#测试环境一键维护]]。清理命令会暂时停止 Hermes 测试会话和共享 service，完成后自动恢复 service。

## 0. 前置条件

在仓库根目录执行：

```powershell
npm.cmd install
npm.cmd run check
docker compose -f compose.yaml -f compose.hermes.yaml build hermes-local hermes-a hermes-b
```

准备一个 Team Memory service 可访问的真实 HTTP embedding 服务。`Dev` 和 `Production` 都不能用 fake embedding。

## 1. 启动 server 基础设施

```powershell
docker compose up --build -d libsql qdrant object-store
```

## 2. 运行 server-local setup

```powershell
npm.cmd run team -- --config config/team-memory.server-local.json setup
```

交互提示里按测试环境输入：

| 提示 | Test 2 输入 |
| --- | --- |
| runtime mode | `Dev` 或 `Production` |
| embedding provider | `http` |
| embedding URL | server runtime 可访问的 HTTP embedding URL |
| embedding model | 当前 embedding 服务中真实存在的模型 |
| libSQL | server-local 测试使用的 libSQL 配置 |
| CAS | object-store 或测试要求的 CAS 配置 |
| Qdrant | server-local Qdrant 地址 |
| recall top-P | 默认 `0.8`；控制 recall 结果覆盖的分数比例，Agent 的 `limit` 仍只是返回数量上限 |
| optional settings | 没有特殊需求就接受默认值 |

通过条件：

- setup 验证 embedding model。
- setup 成功后写 activation record。
- 未激活配置不能启动为有效服务。

## 3. 启动 Team Memory service

```powershell
docker compose up --build -d service
```

健康检查：

```powershell
curl.exe http://127.0.0.1:3000/live
curl.exe http://127.0.0.1:3000/ready
```

## 4. Bootstrap server root admin

```powershell
$env:BOOTSTRAP_ROOT_ENTITY_ID = "root:test2-server"
$env:BOOTSTRAP_USER_ID = "user:test2-admin"
$env:BOOTSTRAP_USER_NAME = "Test 2 Server Admin"
$env:BOOTSTRAP_SESSION_ID = "session:test2-admin"
$env:BOOTSTRAP_SESSION_EXPIRES_AT = "2030-01-01T00:00:00.000Z"
$env:BOOTSTRAP_USER_PASSWORD = "<test server admin password>"
npm.cmd run bootstrap:root-admin -- --config config/team-memory.server-local.json
```

保存返回的 admin token：

```powershell
$env:ADMIN_TOKEN = "<server admin session token>"
```

验证身份：

```powershell
curl.exe -H "Authorization: Bearer $env:ADMIN_TOKEN" http://127.0.0.1:3000/identity
```

## 5. 用 server admin 创建 Hermes client sessions

只有 server admin token 可以做这一步。普通 Hermes agent token 不能配置权限。

```powershell
$env:TEAM_MEMORY_TOKEN = $env:ADMIN_TOKEN
npm.cmd run team -- --config config/team-memory.server-local.json agents onboard agent:test2-hermes-a delegation:test2-hermes-a session:test2-hermes-a 2030-01-01T00:00:00.000Z
npm.cmd run team -- --config config/team-memory.server-local.json agents onboard agent:test2-hermes-b delegation:test2-hermes-b session:test2-hermes-b 2030-01-01T00:00:00.000Z
```

保存返回 token：

```powershell
$env:HERMES_A_TOKEN = "<Hermes A agent token>"
$env:HERMES_B_TOKEN = "<Hermes B agent token>"
```

创建只读 client：

```powershell
npm.cmd run team -- --config config/team-memory.server-local.json agents onboard agent:test2-hermes-readonly delegation:test2-hermes-readonly session:test2-hermes-readonly 2030-01-01T00:00:00.000Z read-only
$env:HERMES_READONLY_TOKEN = "<Hermes read-only agent token>"
```

## 6. 配置 Hermes A 和 Hermes B

Hermes A/B 使用不同 `/root/.hermes` volume，各自都要配置 Hermes 原生设置和 memory provider。

Terminal A：

```powershell
docker compose -f compose.yaml -f compose.hermes.yaml run --rm hermes-a hermes setup
docker compose -f compose.yaml -f compose.hermes.yaml run --rm hermes-a hermes config
docker compose -f compose.yaml -f compose.hermes.yaml run --rm hermes-a hermes memory setup team_memory
docker compose -f compose.yaml -f compose.hermes.yaml run --rm hermes-a hermes memory status
```

Terminal B：

```powershell
docker compose -f compose.yaml -f compose.hermes.yaml run --rm hermes-b hermes setup
docker compose -f compose.yaml -f compose.hermes.yaml run --rm hermes-b hermes config
docker compose -f compose.yaml -f compose.hermes.yaml run --rm hermes-b hermes memory setup team_memory
docker compose -f compose.yaml -f compose.hermes.yaml run --rm hermes-b hermes memory status
```

通过条件：

- 两边 `hermes memory status` 都显示 `Provider: team_memory`。
- 两边 provider 指向 `http://service:3000`。
- 两边身份都是 `root:test2-server` 下的 server-issued session。

## 7. 启动 Hermes A/B 会话

Terminal A：

```powershell
docker compose -f compose.yaml -f compose.hermes.yaml run --rm hermes-a hermes
```

Terminal B：

```powershell
docker compose -f compose.yaml -f compose.hermes.yaml run --rm hermes-b hermes
```

从这里开始保存 A/B 两份 transcript。后续 Test 2 的通过条件必须由 Hermes 对话和日志证明。

## 8. 采集日志

Hermes client logs：

```powershell
docker compose -f compose.yaml -f compose.hermes.yaml run --rm hermes-a sh -lc "tail -n 200 /root/.hermes/team-memory-tool-calls.jsonl"
docker compose -f compose.yaml -f compose.hermes.yaml run --rm hermes-b sh -lc "tail -n 200 /root/.hermes/team-memory-tool-calls.jsonl"
```

Server logs：

```powershell
docker compose logs service
```

最终验收必须同时保留：

- Hermes A transcript。
- Hermes B transcript。
- Hermes A/B tool-call logs。
- service log 中对应请求。
- admin CLI 输出。
- conflict/sync/admin resolution 输出。
