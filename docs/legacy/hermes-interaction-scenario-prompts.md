# Hermes Interaction Workflow Daily-Use Prompts

这份文档是 Hermes 手工回归测试的唯一执行文档。每个 scenario 同时写明：

1. 发送给 Hermes 的自然语言提示词。
2. Agent 应做什么。
3. Hermes 应调用的 Team Memory 工具 JSON schema。
4. 预期调用 payload。
5. 真实调用路径。
6. 预期返回结果。
7. 验收结果。

测试员只把“发送给 Hermes”代码块复制进 Hermes。其他内容用于检查 dev
tool-call log、hook log、Team Memory active view 和返回 JSON。

## Shared Rules For The Whole Run

- Agent-facing capture interface is smaller than the internal memory model.
- Agent chooses operation target, operation type, agent-authored names, semantic
  properties, tags, extra, and relation endpoints.
- Team Memory owns generated ids, root/session identity, commit ids,
  `createdAt`, `updatedAt`, embeddings, BM25/vector projection fields, branch
  refs, resolved endpoint ids, CAS metadata, History records, and ranking /
  confidence signals such as branch `importance`, branch/relation `confidence`,
  and relation `weight`. Until scoring is fully implemented, those fields stay
  empty or `0`.
- Agent-authored `properties.name` must be a durable short memory name. It must
  not be the raw user prompt.
- `properties.name` maps to internal `MemoryEntity.name` / `MemoryEntity.title`
  for entity writes, and to internal `MemoryEntityBranch.title` for branch
  writes. Agent does not send a separate `title`.
- `properties.desc` maps to internal `description` for entity and branch
  writes. For entity writes, it is the L3 summary; for branch writes, it is the
  L2 atomic fact body.
- A branch write identifies its parent entity with `subject`.
- A relation write identifies endpoints with `subject` and `object`, and uses
  `type` for the relation type. It does not include `properties`; relation
  description, workflow order, required-step flags, evidence role, weight, and
  confidence are Team Memory internals.
- Do not use legacy operation names such as `upsert_memory_entity`,
  `create_memory_entity_branch`, or `create_memory_relation` in Hermes
  agent-facing calls.
- Do not use `entityName`, `title`, `relationType`, `source`, `target` endpoint
  objects, `action`, generated ids, root/user/agent identity, timestamps,
  `oldClaim`, `newClaim`, `includeHistory`, `properties.importance`,
  `properties.confidence`, `properties.weight`, relation `properties.desc`,
  relation `properties.role`, relation `properties.ordinal`, relation
  `properties.required`, or top-level `conflict: true`.
- Raw documents are not ordinary semantic capture; they belong in Resource/CAS
  import and ingestion paths.
- Tool-call examples in this file are written as `Tool:` plus `Input JSON:`.
  The JSON block is the exact argument object validated by the tool schema. Do
  not wrap the argument object in `{ "tool": "...", "input": ... }`; that shape
  is only a log envelope when a host chooses to display one.
- `clientMutationId` is not supplied by Hermes/Agent in ordinary capture calls.
  It is generated by Team Memory or by the trusted adapter when a lower-level
  History/CAS command needs idempotency.

## Common Tool Schemas

### team_memory_search

```json
{
  "type": "object",
  "properties": {
    "query": { "type": "string" },
    "limit": { "type": "integer" },
    "layer": { "type": "string", "enum": ["L1", "L2", "L3"] },
    "names": { "type": "array", "items": { "type": "string" } },
    "tagsAny": { "type": "array", "items": { "type": "string" } }
  },
  "required": ["query"],
  "additionalProperties": false
}
```

### team_memory_catalog

```json
{
  "type": "object",
  "properties": {},
  "additionalProperties": false
}
```

### team_memory_capture

```json
{
  "type": "object",
  "properties": {
    "operations": {
      "type": "array",
      "items": {
        "type": "object",
        "properties": {
          "target": {
            "type": "string",
            "enum": [
              "memory_entity",
              "memory_entity_branch",
              "memory_relation"
            ]
          },
          "op": {
            "type": "string",
            "enum": [
              "create",
              "update",
              "refresh",
              "update_metadata",
              "replace"
            ]
          },
          "subject": {
            "oneOf": [
              { "type": "string" },
              { "$ref": "#/$defs/relationEndpoint" }
            ]
          },
          "object": { "$ref": "#/$defs/relationEndpoint" },
          "type": {
            "type": "string",
            "enum": [
              "has",
              "depends_on",
              "relates_to",
              "refers_to",
              "contradicts",
              "supersedes",
              "next_is"
            ]
          },
          "properties": {
            "type": "object",
            "properties": {
              "name": { "type": "string" },
              "desc": { "type": "string" },
              "tags": { "type": "array", "items": { "type": "string" } },
              "status": { "type": "string" },
              "extra": { "type": "object" }
            },
            "additionalProperties": false
          }
        },
        "required": ["target", "op"],
        "additionalProperties": false,
        "allOf": [
          {
            "if": {
              "properties": { "target": { "const": "memory_relation" } },
              "required": ["target"]
            },
            "then": {
              "not": { "required": ["properties"] }
            }
          }
        ]
      }
    }
  },
  "required": ["operations"],
  "additionalProperties": false,
  "$defs": {
    "relationEndpoint": {
      "type": "object",
      "properties": {
        "target": {
          "type": "string",
          "enum": [
            "memory_entity",
            "memory_entity_branch",
            "resource",
            "resource_chunk"
          ]
        },
        "name": { "type": "string" },
        "parent": { "type": "string" }
      },
      "required": ["target", "name"],
      "additionalProperties": false
    }
  }
}
```

Required and optional fields by operation:

| target | op | required | optional |
| --- | --- | --- | --- |
| `memory_entity` | `create` | `properties.name`, `properties.desc` | `properties.tags`, `properties.status`, `properties.extra` |
| `memory_entity` | `update` | `properties.name` | `properties.desc`, `properties.tags`, `properties.status`, `properties.extra` |
| `memory_entity` | `refresh` | `properties.name`, `properties.desc` | `properties.tags`, `properties.status`, `properties.extra` |
| `memory_entity_branch` | `create` | `subject`, `properties.name`, `properties.desc` | `properties.tags`, `properties.status`, `properties.extra` |
| `memory_entity_branch` | `update_metadata` | `subject`, `properties.name` | `properties.tags`, `properties.status`, `properties.extra` |
| `memory_relation` | `create` | `type`, `subject`, `object` | none |
| `memory_relation` | `replace` | `type`, `subject`, `object` | none |

For `memory_entity_branch`, `subject` is the parent entity's agent-authored
name. For relation endpoints, use:

```json
{
  "target": "memory_entity_branch",
  "name": "Confirm latest ticket summaries",
  "parent": "Riverfront release checklist"
}
```

`parent` is required when endpoint `target` is `memory_entity_branch`.
Endpoint `target` is the endpoint object kind; it is not the operation-level
`target` that selects which memory object family the operation writes.

### team_memory_lifecycle_log

```json
{
  "type": "object",
  "properties": {
    "limit": { "type": "integer", "default": 50 }
  },
  "additionalProperties": false
}
```

## Scenario 0: Dev Log Preflight

发送给 Hermes：

```text
/new
我准备做一轮长期记忆回归测试。开始前，帮我确认一下你现在的长期记忆功能是否可用，以及有没有可查看的调试日志。请用普通用户能看懂的话告诉我：当前记忆 provider、当前模式、会话是否有效、日志在哪里。
```

Agent:

1. Check whether Team Memory provider is available.
2. Check current Hermes session and tool-call logging.
3. Do not write memory.

Expected call:

Tool:

```text
team_memory_lifecycle_log
```

Input JSON:

```json
{ "limit": 10 }
```

Real path:

1. Hermes calls `team_memory_lifecycle_log`.
2. Plugin reads hook log and dev tool-call log.
3. Plugin returns log paths, recent entries, and whether tool-call logging is on.

Memory result:

```json
{
  "hookLogFile": "/root/.hermes/team-memory-hooks.jsonl",
  "toolCallLogFile": "/root/.hermes/team-memory-tool-calls.jsonl",
  "toolCallLogEnabled": true,
  "entries": [],
  "toolCallEntries": []
}
```

验收结果:

- 应说明 provider 是 `team_memory`。
- 应说明当前模式是 local 或 http。
- 应给出日志路径。
- 不应调用 `team_memory_capture`。

## Scenario 1: Provider Availability And Identity

发送给 Hermes：

```text
我不想把今天的项目背景记错地方。你现在用的是哪套长期记忆？它是本地模式还是服务端模式？你能看到的记忆空间大概是什么？顺便告诉我你能不能保存接下来我说的项目偏好。
```

Agent:

1. Call catalog to confirm provider/session visibility.
2. Report provider mode and memory capability without exposing root ids.
3. Do not write memory.

Expected call:

Tool:

```text
team_memory_catalog
```

Input JSON:

```json
{}
```

Real path:

1. `team_memory_catalog` calls `HermesTeamMemoryProvider.catalog()`.
2. Provider calls `memory.catalog`.
3. Gateway authenticates the trusted token.
4. `PermissionRouter` authorizes `read:memory_entity`.
5. Gateway returns L3 directory names and tags only.

Memory result:

```json
{
  "rootName": "test1-local",
  "branchRef": "main",
  "entities": [],
  "tags": []
}
```

验收结果:

- 应报告 Team Memory provider 可用。
- 不应输出 `rootEntityId`。
- 不应写入记忆。

## Scenario 4: Initial Recall Before Answering

发送给 Hermes：

```text
/new
我昨天跟你说的那个“流失预警试点”到底叫什么正式项目名？它为什么会跟 OpenClaw 扯上关系？我现在要写周报，先帮我用两句话回忆一下。
```

Agent:

1. Receive the user prompt.
2. Call `team_memory_search` with natural-language query and optional filters.
3. Answer from returned branches, chunks, resources, and relations.
4. Do not write memory.

Expected call:

Tool:

```text
team_memory_search
```

Input JSON:

```json
{
  "query": "流失预警试点 正式项目名 OpenClaw 客服工单摘要 Riverfront",
  "limit": 5,
  "layer": "L2",
  "tagsAny": ["project:riverfront"]
}
```

Real path:

1. `team_memory_search` -> `HermesTeamMemoryProvider.search`.
2. Provider calls `memory.search` because filters are supplied.
3. Gateway authenticates token and rejects identity override fields.
4. `PermissionRouter` authorizes `search:memory_entity`.
5. `MemoryRetrievalAdapter` searches BM25, vectors, SQL active view, and
   relation graph.
6. Results are ranked and returned.

Memory result:

```json
{
  "rootEntityId": "00000000-0000-4000-8000-000000000001",
  "branchRef": "main",
  "items": [
    {
      "kind": "entity",
      "entity": { "id": "entity:riverfront" },
      "branch": {
        "id": "branch:riverfront-naming-preference",
        "title": "Riverfront naming preference",
        "description": "Riverfront is the formal project name for the Nova CRM customer churn warning pilot. In daily conversation, call it 流失预警试点; in report titles, still write Riverfront.",
        "tags": ["project:riverfront", "naming"]
      },
      "score": 0.96,
      "origin": "hybrid"
    },
    {
      "kind": "entity",
      "entity": { "id": "entity:riverfront" },
      "branch": {
        "id": "branch:riverfront-receives-openclaw-ticket-summaries",
        "title": "Riverfront receives OpenClaw ticket summaries",
        "description": "Riverfront is related to OpenClaw because OpenClaw is responsible for pushing customer support ticket summaries to Riverfront.",
        "tags": ["project:riverfront", "openclaw"]
      },
      "score": 0.93,
      "origin": "hybrid"
    }
  ]
}
```

验收结果:

- 回答前必须有 search/recall 调用。
- 应回答正式名是 Riverfront。
- 应说明日常叫“流失预警试点”。
- 应说明它和 OpenClaw 的关系来自客服工单摘要。
- 不应依赖短期上下文。
- 不应调用 `team_memory_capture`。

1. Search Riverfront/OpenClaw.
2. Use returned branches and relations.
3. Do not write memory.

Expected call:

Tool:

```text
team_memory_search
```

```json
{
  "query": "Riverfront OpenClaw 关系 客服工单摘要",
```
