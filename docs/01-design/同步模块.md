# 同步模块

同步模块负责 Cloud authority 和 Local authorized working replica 之间的数据交换。它不定义记忆对象语义；语义对象见[[记忆模块.md#MemoryEntity]]。它也不绕过 RBAC；授权边界见[[RBAC模块.md#PermissionRouter]]。

代码出处：`src/sync/working-replica.ts`、`src/sync/pending.ts`、`src/sync/index.ts`、`src/memory/sync.ts`、`src/adapters/filesystem/local-replica.ts`。

本模块术语：

| 名称 | 含义 |
| --- | --- |
| Cloud authority | 服务端权威状态，拥有最终 History、CAS、RBAC 和 active view。 |
| Local authorized working replica | 本地可读写副本，只包含当前授权范围内的对象，并可叠加本地 pending 变更。 |
| active view | 从 History commit 投影出来的当前可见状态。 |
| pending overlay | 本地尚未被 Cloud 接受的变更叠加层；它影响本地视图，但不是云端事实。 |
| watermark | Cloud authority 的单调进度标记，用于增量 pull/push 冲突判断。 |

## AuthorizedSnapshot

`AuthorizedSnapshot` 是 Cloud active view 的授权子集，只包含当前 user/agent/root/task scope 可读写的对象。

必须字段：`rootEntityId`、`branchRef`、`watermark`、`entities`、`branches`、`relations`、`resources`。

可选字段：`deniedCounts`、`policyVersion`。

字段含义：

| 字段 | 含义 |
| --- | --- |
| `rootEntityId` | snapshot 所属记忆 root。 |
| `branchRef` | snapshot 对应的逻辑分支，例如 `main`。 |
| `watermark` | 生成 snapshot 时 Cloud authority 的同步进度。 |
| `entities` | 授权可见的 `MemoryEntity` 列表。 |
| `branches` | 授权可见的 `MemoryEntityBranch` 列表。 |
| `relations` | 授权可见且两端可解析的 `MemoryRelation` 列表。 |
| `resources` | 授权可见的 L1 resource 摘要。 |
| `deniedCounts` | 因权限过滤被排除的对象计数，只用于诊断，不泄露对象内容。 |
| `policyVersion` | 生成 snapshot 时使用的 RBAC policy 版本。 |

示例结果：

```json
{
  "rootEntityId": "root:riverfront",
  "branchRef": "main",
  "watermark": "history:42",
  "entities": [{ "name": "Riverfront" }],
  "branches": [{ "parent": "Riverfront", "name": "Riverfront naming preference" }],
  "relations": [{ "type": "relates_to", "subject": "Riverfront", "object": "OpenClaw" }],
  "resources": []
}
```

预期结果：Local 只获得授权对象；未授权对象不会因 relation expansion 泄露。

## WorkingReplica

`WorkingReplica` 是本地授权工作副本，用于离线读、局部写、pending overlay 和 later sync。

必须字段：`snapshot`、`pendingOperations`、`lastSyncedWatermark`。

可选字段：`conflicts`、`localMetadata`。

字段含义：

| 字段 | 含义 |
| --- | --- |
| `snapshot` | 最近一次从 Cloud 拉取的授权 active view。 |
| `pendingOperations` | 本地已接受、尚未 push 成功的变更。 |
| `lastSyncedWatermark` | 本地已确认同步到的 Cloud watermark。 |
| `conflicts` | Cloud push/pull 发现的同步冲突，不等同于语义矛盾。 |
| `localMetadata` | 本地缓存、诊断、文件路径等非权威元数据。 |

示例调用：

```json
{
  "operation": "load",
  "rootEntityId": "root:riverfront",
  "branchRef": "main"
}
```

预期结果：加载本地 snapshot，并叠加未同步 pending operations。

## PendingOperation

`PendingOperation` 是 local 尚未被 Cloud authority 接受的变更。它可来源于[[记忆模块.md#AgentFacingCaptureInput]] 或 resource revision。

系统负责字段：`pendingId`、`clientMutationId`、`createdAt`、`baseWatermark`、`status`。

Agent 负责字段：仍然只负责 capture input；不直接创建 pending metadata。

字段含义：

| 字段 | 含义 |
| --- | --- |
| `pendingId` | 本地 pending 变更 id。 |
| `clientMutationId` | 幂等键，由本地 adapter/系统生成。 |
| `createdAt` | pending 变更进入本地队列的时间。 |
| `baseWatermark` | 创建 pending 时本地所基于的 Cloud watermark。 |
| `status` | pending 当前状态，例如 queued/applied/conflict。 |
| `operationTargets` | 本次变更触及的对象类别，用于授权和冲突诊断。 |

示例结果：

```json
{
  "pendingId": "pending:riverfront:1",
  "clientMutationId": "generated-by-local-adapter",
  "baseWatermark": "history:42",
  "status": "queued",
  "operationTargets": ["memory_entity_branch"]
}
```

预期结果：本地可见 pending overlay；同步成功后 pending 标记为 applied。

## SyncPull

Pull 从 Cloud 拉取授权事件和 active view 更新。

必须字段：`rootEntityId`、`branchRef`、`sinceWatermark`。

可选字段：`limit`。

字段含义：

| 字段 | 含义 |
| --- | --- |
| `rootEntityId` | 要拉取的记忆 root。 |
| `branchRef` | 要拉取的逻辑分支。 |
| `sinceWatermark` | 只拉取该 watermark 之后的变化。 |
| `limit` | 单次返回事件数量上限。 |
| `events` | Cloud 返回的增量事件列表。 |

示例调用：

```json
{
  "rootEntityId": "root:riverfront",
  "branchRef": "main",
  "sinceWatermark": "history:42"
}
```

预期结果：

```json
{
  "watermark": "history:45",
  "events": [
    { "kind": "commit_applied", "commitId": "commit:remote-43" }
  ]
}
```

## SyncPush

Push 把 pending operations 提交给 Cloud authority。Cloud 重新做权限和冲突校验。

必须字段：`pendingOperations`、`baseWatermark`。

可选字段：`dryRun`。

字段含义：

| 字段 | 含义 |
| --- | --- |
| `pendingOperations` | 要提交的本地 pending id 或 pending payload 列表。 |
| `baseWatermark` | 这些 pending 基于的 Cloud watermark。 |
| `dryRun` | 只做权限/冲突检查，不真正提交。 |
| `appliedPendingIds` | 已被 Cloud 接受的 pending id。 |
| `newWatermark` | push 成功后的 Cloud watermark。 |
| `conflicts` | 未能自动接受的冲突列表。 |

示例调用：

```json
{
  "baseWatermark": "history:42",
  "pendingOperations": ["pending:riverfront:1"]
}
```

预期结果：

```json
{
  "status": "applied",
  "appliedPendingIds": ["pending:riverfront:1"],
  "newWatermark": "history:46"
}
```

冲突结果：

```json
{
  "status": "conflict",
  "conflicts": [
    { "pendingId": "pending:riverfront:1", "conflictKey": "memory_entity_branch:Riverfront:Weekly report audience" }
  ]
}
```

## SyncConflictResolution

同步冲突解决使用[[操作记录模块.md#ConflictRecord]]，不靠 Agent 自然语言猜测。

字段含义：

| 字段 | 含义 |
| --- | --- |
| `operation` | 冲突处理动作，例如 `resolve_conflict`。 |
| `conflictKey` | 冲突对象的稳定定位键，由系统生成。 |
| `strategy` | 解决策略，例如 keep_local、keep_remote、manual。 |
| `newCommitId` | 解决冲突后生成的 commit id。 |

示例调用：

```json
{
  "operation": "resolve_conflict",
  "conflictKey": "memory_entity_branch:Riverfront:Weekly report audience",
  "strategy": "keep_remote"
}
```

预期结果：

```json
{
  "status": "resolved",
  "strategy": "keep_remote",
  "newCommitId": "commit:resolve-conflict"
}
```

验收：解决冲突后重新投影 active view；pending 队列不保留已解决项。
