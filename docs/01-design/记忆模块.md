# 记忆模块

本模块定义 Team Memory 的语义对象、Agent 写入契约、检索契约和对象间关系。根据[[记忆模块.md#AgentFacingCaptureInput]]，Agent 只负责说明要改哪个对象、执行什么操作、写入哪些语义属性；`id`、`createdAt`、`updatedAt`、`clientMutationId`、embedding、BM25/vector 投影、commit、endpoint id 解析、`importance`、`confidence`、`weight` 等排序/评分信号都由记忆模块负责。当前评分计算尚未完整实现时，这些系统字段留空或由系统填 `0`。

代码出处：`src/contracts/memory.ts`、`src/contracts/schema.ts`、`src/memory/contracts.ts`、`src/memory/authority.ts`、`src/memory/stores.ts`、`src/memory/retrieval.ts`、`src/memory/projector.ts`、`src/memory/projection-worker.ts`、`src/memory/sync.ts`、`src/memory/pending.ts`、`src/memory/index.ts`。

相关模块：RBAC 授权见[[RBAC模块.md#PermissionRouter]]；提交历史和幂等见[[操作记录模块.md#MemoryCommit]]；本地副本见[[同步模块.md#WorkingReplica]]；资源证据见[[资源摄取模块.md#Resource]]；Hermes 暴露的工具面见[[运行时适配器模块.md#TeamMemoryGateway]]。

## AgentFacingCaptureInput

Agent-facing capture input 是稳定的普通工具参数，不包含 host log envelope。

必须字段：`operations`。

可选字段：无。`clientMutationId` 不由 Agent 提供；低层 History/CAS 需要幂等键时，由 Team Memory 或 trusted adapter 生成。

字段含义：

| 字段 | 含义 |
| --- | --- |
| `operations` | 本次 capture 要提交的一组原子语义操作。一个 `operations[]` batch 映射为一个 History commit。 |
| `clientMutationId` | 幂等键；普通 Agent-facing capture 禁止传入，由 Team Memory 或 trusted adapter 生成。 |

示例调用：

```json
{
  "operations": [
    {
      "target": "memory_entity",
      "op": "create",
      "properties": {
        "name": "Riverfront",
        "desc": "Riverfront is the Nova CRM customer churn warning pilot.",
        "tags": ["project:riverfront"]
      }
    }
  ]
}
```

预期结果：

```json
{
  "status": "captured",
  "outcome": "success",
  "commitIds": ["commit:generated-by-memory-system"],
  "items": [
    { "kind": "memory_entity", "name": "Riverfront" }
  ]
}
```

不合法示例：

```json
{
  "clientMutationId": "scenario-3-riverfront-seed",
  "operations": []
}
```

预期结果：文档验收应判定为错误，因为 Agent 不负责提供 `clientMutationId`。

## MemoryCaptureOperation

每个 `operations[]` item 是一个原子操作；同一次 capture 调用里的所有 item 组成一个 History commit。根据[[操作记录模块.md#MemoryOperation]]，commit 记录操作顺序和结果，但 Agent 不直接构造 commit。

通用字段：

| 字段 | 必须 | 负责方 | 说明 |
| --- | --- | --- | --- |
| `target` | 是 | Agent | `memory_entity`、`memory_entity_branch`、`memory_relation` |
| `op` | 是 | Agent | `create`、`update`、`refresh`、`update_metadata`、`replace` 中适用于 target 的值 |
| `properties` | 按 target | Agent | Agent 可写语义属性；不包含 `importance`、`confidence`、`weight` 等系统排序/评分字段 |
| `subject` | 按 target | Agent | branch 的父实体名，或 relation 的主语 endpoint |
| `object` | relation 必须 | Agent | relation 的宾语 endpoint |
| `type` | relation 必须 | Agent | relation type，映射到内部 `MemoryRelation.relationType` |

系统字段：operation id、resolved source/target ids、commit id、timestamps、root/session/user identity、embedding、projection metadata、branch/relation ranking metadata。

Agent-facing `properties` 字段含义：

| 字段 | 适用对象 | 含义 |
| --- | --- | --- |
| `properties.name` | `memory_entity`、`memory_entity_branch` | Agent 为对象起的稳定短名。实体写入时映射内部 `MemoryEntity.name`/`title`；branch 写入时映射内部 `MemoryEntityBranch.title`。不能直接使用用户整句原始 prompt。 |
| `properties.desc` | `memory_entity`、`memory_entity_branch` | Agent 写入的语义描述，映射内部 `description`。对 entity 是 L3 摘要；对 branch 是 L2 原子事实正文。 |
| `properties.tags` | `memory_entity`、`memory_entity_branch` | Agent 可写的分类标签，用于检索过滤和目录归类。 |
| `properties.status` | `memory_entity`、`memory_entity_branch` | Agent 可写的非破坏性状态标记；删除、冲突裁决等管理动作仍由 RBAC 控制的系统路径处理。 |
| `properties.extra` | `memory_entity`、`memory_entity_branch` | 对象自身的结构化补充信息。依赖、引用、顺序、冲突、包含、替代等对象间关系必须写成 `MemoryRelation`，不能塞进 `extra`。 |

`MemoryRelation` 的 Agent-facing 写入只包含 `type`、`subject`、`object`。`properties.desc`、`properties.role`、`properties.ordinal`、`properties.required`、`properties.importance`、`properties.confidence`、`properties.weight` 都不是 Agent-facing relation 字段。`MemoryEntityBranch.importance`/`confidence` 和 `MemoryRelation.weight`/`confidence` 由记忆模块根据去重、重复提及、recency、检索表现等信号计算；当前未实现时保持空值或 `0`。workflow 顺序、是否必经、证据角色等 relation 派生属性也由记忆模块内部维护，不让 Agent 手填。

## MemoryEntity

`MemoryEntity` 是 L3 稳定身份和摘要，不是原子事实。Agent 通过 `properties.name` 指定实体名；该字段同时映射内部 `name`/`title`，不能把用户整句原始提示词当 name。

必须字段：

| op | 必须字段 | 可选字段 |
| --- | --- | --- |
| `create` | `properties.name`, `properties.desc` | `properties.tags`, `properties.status`, `properties.extra` |
| `update` | `properties.name` | `properties.desc`, `properties.tags`, `properties.status`, `properties.extra` |
| `refresh` | `properties.name`, `properties.desc` | `properties.tags`, `properties.status`, `properties.extra` |

创建示例：

```json
{
  "operations": [
    {
      "target": "memory_entity",
      "op": "create",
      "properties": {
        "name": "OpenClaw",
        "desc": "OpenClaw provides customer support ticket summaries used by Riverfront.",
        "tags": ["openclaw"]
      }
    }
  ]
}
```

预期结果：

```json
{
  "status": "captured",
  "items": [
    {
      "kind": "memory_entity",
      "name": "OpenClaw",
      "desc": "OpenClaw provides customer support ticket summaries used by Riverfront."
    }
  ]
}
```

更新摘要示例：

```json
{
  "operations": [
    {
      "target": "memory_entity",
      "op": "update",
      "properties": {
        "name": "Riverfront",
        "desc": "Riverfront is the formal report name for the Nova CRM churn warning pilot."
      }
    }
  ]
}
```

预期结果：更新 L3 摘要，不因摘要变化自动创建 `MemoryEntityBranch`。

## MemoryEntityBranch

`MemoryEntityBranch` 是 L2 原子事实。`subject` 是父 `MemoryEntity` 的 Agent-authored name；`properties.name` 是 Agent 为该原子事实起的短名，映射内部 branch title。

当 Agent 创建 `memory_entity_branch` 时，`subject` 只负责指出父 `MemoryEntity`。
记忆模块必须在同一 commit 内自动维护 `MemoryEntity has MemoryEntityBranch`
关系边；Agent 不需要、也不应该手写 `memory_relation create type:"has"`。这条
`has` 边是 branch 创建的不变量，用于 workflow/relation graph 检索。命中重复项
过滤时不创建新 branch；系统只更新已有 branch 的 recency/importance/confidence
等 metadata，必要时可补齐缺失的系统维护边。同一父 entity 下，标准化后的
branch title 精确相同也视为重复信号，即使 description embedding 没有超过语义
dedupe threshold。

如果同一父 entity 下存在相似度较高、但低于 dedupe threshold 的 branch，
记忆模块不能自动判定取代、冲突或普通关联；它必须把候选关键信息返回给
Agent，例如 `name`、`desc`、`tags`、`extra`、`similarity`，并附带推荐动作：
由 Agent 根据语义再写一条 `memory_relation create`，关系类型可为
`relates_to`、`supersedes` 或 `contradicts`。这类返回是写入结果中的提示，
不是自动创建的关系边。

必须字段：

| op | 必须字段 | 可选字段 |
| --- | --- | --- |
| `create` | `subject`, `properties.name`, `properties.desc` | `properties.tags`, `properties.status`, `properties.extra` |
| `update_metadata` | `subject`, `properties.name` | `properties.tags`, `properties.status`, `properties.extra` |

创建示例：

```json
{
  "operations": [
    {
      "target": "memory_entity_branch",
      "op": "create",
      "subject": "Riverfront",
      "properties": {
        "name": "Riverfront naming preference",
        "desc": "Call Riverfront 流失预警试点 in daily conversation, but use Riverfront in formal report titles.",
        "tags": ["project:riverfront", "naming"]
      }
    }
  ]
}
```

预期结果：

```json
{
  "status": "captured",
  "items": [
    {
      "kind": "memory_entity_branch",
      "parent": "Riverfront",
      "name": "Riverfront naming preference"
    }
  ]
}
```

重复事实示例：

```json
{
  "operations": [
    {
      "target": "memory_entity_branch",
      "op": "update_metadata",
      "subject": "Riverfront",
      "properties": {
        "name": "Riverfront naming preference",
        "tags": ["project:riverfront", "naming", "seen-again"]
      }
    }
  ]
}
```

预期结果：不改写原 branch 正文；Agent 只更新可写 metadata，记忆模块自己更新 recency/importance/confidence 等排序元数据。当前评分未实现时，`importance`/`confidence` 留空或为 `0`。

## MemoryRelation

`MemoryRelation` 是 L2 语义边。Agent 写入时字段名是 `type`；内部存储可叫 `relationType`，但这不是 Agent-facing 写入字段。

必须字段：

| op | 必须字段 | 可选字段 |
| --- | --- | --- |
| `create` | `type`, `subject`, `object` | 无 |
| `replace` | `type`, `subject`, `object` | 无 |

endpoint 结构：

```json
{
  "target": "memory_entity_branch",
  "name": "Confirm latest ticket summaries",
  "parent": "Riverfront release checklist"
}
```

Relation endpoint 字段含义：

| 字段 | 含义 |
| --- | --- |
| `target` | endpoint 指向的对象类型，可为 `memory_entity`、`memory_entity_branch`、`resource`、`resource_chunk`。这里的 `target` 是 endpoint kind，不是 `operations[]` item 的操作目标。 |
| `name` | Agent 可见的对象名；对 `resource`/`resource_chunk` 可是已导入对象的可见名或稳定标识。Agent 不传内部 UUID。 |
| `parent` | 仅 `target = memory_entity_branch` 时必填，表示父 `MemoryEntity` 的 Agent-authored name，用来消除同名 branch 歧义。 |

`MemoryRelation.weight` 和 `MemoryRelation.confidence` 是记忆模块的内部排序/置信度信号，不由 Agent 传入。Agent 创建或替换 relation 时只描述边的类型和两端 endpoint；关系说明、工作流顺序、必需性、证据角色等派生属性由记忆模块内部维护。

实体关系示例：

```json
{
  "operations": [
    {
      "target": "memory_relation",
      "op": "create",
      "type": "relates_to",
      "subject": { "target": "memory_entity", "name": "Riverfront" },
      "object": { "target": "memory_entity", "name": "OpenClaw" }
    }
  ]
}
```

预期结果：

```json
{
  "status": "captured",
  "items": [
    {
      "kind": "memory_relation",
      "type": "relates_to",
      "subject": "Riverfront",
      "object": "OpenClaw"
    }
  ]
}
```

流程顺序示例：

```json
{
  "operations": [
    {
      "target": "memory_relation",
      "op": "create",
      "type": "next_is",
      "subject": {
        "target": "memory_entity_branch",
        "name": "Confirm latest ticket summaries",
        "parent": "Riverfront release checklist"
      },
      "object": {
        "target": "memory_entity_branch",
        "name": "Check configuration changes",
        "parent": "Riverfront release checklist"
      }
    }
  ]
}
```

预期结果：检索 workflow 时 relation packing 按 `next_is` 链和记忆模块内部顺序展开流程。

冲突示例：

```json
{
  "operations": [
    {
      "target": "memory_relation",
      "op": "create",
      "type": "contradicts",
      "subject": {
        "target": "memory_entity_branch",
        "name": "Corrected weekly report audience",
        "parent": "Riverfront"
      },
      "object": {
        "target": "memory_entity_branch",
        "name": "Old weekly report audience",
        "parent": "Riverfront"
      }
    }
  ]
}
```

预期结果：只有显式 `memory_relation create type:"contradicts"` 才产生冲突关系；相似文本或不同值本身不能自动创建冲突。

## MemoryCrudWritePath

Memory CRUD/write path 是 Agent-facing capture 到 History commit、active view、检索索引的完整写入链路。代码里的主要入口是 `TeamMemoryGateway.prepareStructuredMemoryOperations`、`InMemoryMemoryAuthority.appendCommit`、`InMemoryMemoryAuthority.applyOperation`、`HistoryMemoryProjectionWorker.project` 和 `MemoryProjector.project`。

链路字段含义：

| 名称 | 含义 |
| --- | --- |
| Agent-facing operation | Agent 提交的稳定语义操作。canonical 文档使用 `target`、`op`、`subject`、`object`、`type`、`properties` 表达；runtime 里仍有 `upsert_memory_entity`、`create_memory_entity_branch`、`create_memory_relation` 等兼容操作名。 |
| `MemoryOperationInput` | History 层内部操作枚举，定义在 `src/contracts/history.ts`。它是系统内部 commit payload，不是 Agent-facing schema。 |
| `MemoryWriteCommand` | 授权后的写命令，带 trusted `rootEntityId`、`branchRef`、actor、commit id 和 operation 列表。Agent 不传这些 trusted 字段。 |
| active view | `authority` 把 commit replay 后得到的当前可读状态，包含 active/tombstoned 的实体、branch、relation、resource、chunk。 |
| projection write | `projection-worker` 从内部 operation 派生出的 vector point、BM25 document、relation-store upsert/tombstone 操作。 |

Agent-facing CRUD 语义：

| 行为 | Agent-facing 写法 | 内部 operation | 代码行为 |
| --- | --- | --- | --- |
| Create/Upsert entity | `memory_entity create`，必须有 `properties.name`、`properties.desc` | `create_entity` 或 `update_entity` | runtime 按可见 name 精确查找；不存在则创建 L3 entity，存在则更新同一个 entity。不会因为摘要变化自动创建 branch。 |
| Read catalog | `MemoryCatalogInput` | 无写操作 | 读取可见 L3 目录、tag 汇总和当前 root 显示名。 |
| Read recall/search | `MemorySearchInput` | 无写操作 | 走 `MemoryRetrievalAdapter`，先授权再候选生成、过滤、rerank。 |
| Update entity | `memory_entity update` 或 `refresh` | `update_entity` | 替换 entity summary/metadata/embedding；不写 L2 原子事实。 |
| Create branch | `memory_entity_branch create` | `create_entity_branch` + system `create_relation(type:"has")`，或命中重复时 `update_entity_branch_metadata` | branch 是 L2 原子事实。runtime 先检查同一 entity 下是否已有同名 branch，再为 branch description 生成 embedding 并与同一 entity 下已有 branch 比相似度；同名或超过 dedupe threshold 时视为重复提及，只更新系统 metadata，不改写事实正文；低于阈值才创建新 branch，并自动创建父 entity 到新 branch 的 `has` 边。Agent 不手写这条边。 |
| Update branch metadata | `memory_entity_branch update_metadata` | `update_entity_branch_metadata` | 只更新 tags/status/extra 等 metadata；正文事实不被 merge。`importance`、`confidence` 属系统排序字段，普通 Agent 不传。 |
| Create relation | `memory_relation create`，只传 `type`、`subject`、`object` | `create_relation` | runtime 解析 endpoint 可见名到内部 id；authority 验证两端对象存在后写入 relation。Agent 不传 relation role/order/required/weight/confidence。 |
| Replace relation | `memory_relation replace`，只传 `type`、`subject`、`object` | `replace_relation` 展开为 `tombstone_relation` + `create_relation` | authority 的 `expandOperation` 把 replace 拆成 tombstone 旧 relation 和 create 新 relation；projection 同步 tombstone relation store 旧 id 并 upsert 新 relation。 |
| Delete/tombstone | 普通 Agent-facing capture 不暴露删除快捷键 | `tombstone_entity`、`tombstone_entity_branch`、`tombstone_relation`、`tombstone_resource` | 删除是 RBAC 控制的管理动作。tombstone entity 会把该 entity 置为 tombstoned，并 tombstone 涉及它的 relation；tombstone root entity 会级联 tombstone 当前 root 下所有 entity、branch、relation、resource、chunk。 |

内部 `MemoryOperationInput` 字段含义：

| operation kind | 字段 | 含义 |
| --- | --- | --- |
| `create_entity` | `entity` | 完整 `MemoryEntity` 快照，含系统生成 id、时间、embedding/status。 |
| `update_entity` | `targetId`, `entity` | `targetId` 是被替换的 entity id；`entity` 是更新后的完整快照。 |
| `create_entity_branch` | `branch` | 完整 `MemoryEntityBranch` 快照；写入后 entity 的 `currentBranchId` 指向该 branch；runtime 同 commit 追加系统维护的 `has` relation。 |
| `update_entity_branch_metadata` | `targetId`, `branch` | `targetId` 是被更新 branch id；`branch` 是更新后的完整 metadata 快照。 |
| `create_relation` | `relation` | 完整 `MemoryRelation` 快照；authority 会验证 source/target endpoint 已存在。 |
| `replace_relation` | `previousRelationId`, `replacementOperationId`, `replacement` | 系统内部复合操作，表示 tombstone 旧 relation 并创建 replacement。 |
| `tombstone_*` | `targetId` | 被 tombstone 的 resource/entity/branch/relation id。 |
| `create_resource`、`create_resource_chunk`、`revise_resource` | resource/chunk/revision 字段 | 资源摄取写入，详见[[资源摄取模块.md#Resource]] 和[[资源摄取模块.md#ResourceRevision]]。 |
| `revert_commit`、`resolve_conflict` | commit/conflict id 字段 | 操作记录和冲突裁决动作，详见[[操作记录模块.md#RestoreAndRedo]] 和[[操作记录模块.md#ConflictRecord]]。 |

投影规则：

| 内部 operation | Projection 行为 |
| --- | --- |
| `create_entity` / `update_entity` | upsert `memory_entities` vector point；若 entity 没有 embedding，projection-worker 用 name/title、description、tags、status 生成 embedding。 |
| `create_entity_branch` / `update_entity_branch_metadata` | upsert `memory_entity_branches` vector point；`importance`/`confidence` 缺失时 projection payload 写 `0`。branch create 同 commit 还会投影系统维护的 `has` relation。 |
| `create_relation` | upsert relation store。 |
| `replace_relation` | upsert replacement，并 tombstone `previousRelationId`。 |
| `tombstone_relation` | tombstone relation store 对应 id。 |
| `tombstone_entity` | 移除 entity vector point，并按 entityId 过滤移除 branch vector points。 |
| `tombstone_entity_branch` | 移除 branch vector point。 |
| `tombstone_resource` | 按 resourceId 过滤移除 resource chunk vector points。 |
| `create_resource_chunk` | upsert `resource_chunks` vector point，并写入 BM25 document。 |

## MemorySearchInput

检索必须先授权。根据[[RBAC模块.md#PermissionRouter]]，`search:memory_entity` 通过后才进入检索。

必须字段：`query`。

可选字段：`limit`、`layer`、`names`、`tagsAny`。

字段含义：

| 字段 | 含义 |
| --- | --- |
| `query` | 自然语言检索请求。 |
| `limit` | 返回候选数量硬上限。最终结果数量先由配置的 top-P 分数覆盖率决定，再取 `min(topPCount, limit)`。 |
| `layer` | 检索层级：`L3` 查实体摘要，`L2` 查原子事实和 relation-packed context，`L1` 查原始 resource/chunk。 |
| `names` | 用 Agent 可见的 `MemoryEntity` 名称缩小检索范围，不是内部 id。 |
| `tagsAny` | 至少匹配其中一个 tag 的过滤条件；每个值都必须原样复制自当前 `memory.catalog` 返回的可见 `tags`，不能把推断关键词当 tag。 |

如果 catalog 没有合适的可见 tag，使用 `names` 或自然语言 `query` 缩小范围。若 `tagsAny` 至少包含一个可见 tag，检索会过滤未知值、继续按可见 tag 检索，并在 `warnings` 返回结构化 `unknownTags`；若全部值都未知，则返回 `validation_failed`，不会执行普通检索并返回空 `items`。

示例调用：

```json
{
  "query": "Riverfront OpenClaw support ticket summaries",
  "limit": 10,
  "layer": "L2",
  "tagsAny": ["project:riverfront", "openclaw"]
}
```

预期结果：

```json
{
  "rootEntityId": "session-root",
  "branchRef": "main",
  "items": [
    {
      "kind": "entity",
      "entity": { "name": "Riverfront" },
      "branch": {
        "name": "Riverfront receives OpenClaw ticket summaries",
        "desc": "OpenClaw pushes support ticket summaries to Riverfront."
      },
      "relations": [
        { "type": "relates_to", "object": "OpenClaw" }
      ],
      "score": 1,
      "origin": "cloud_active"
    }
  ]
}
```

## MemoryCatalogInput

## MemoryRetrievalQuery

`MemoryRetrievalQuery` 是内部检索 query union；Agent-facing 的 `MemorySearchInput` 只暴露 recall 所需字段，其他 query kind 是 adapter/测试/内部流程使用。

字段含义：

| query kind | 字段 | 含义 |
| --- | --- | --- |
| `recall` | `text`, `limit`, `layer`, `names`, `tagsAny` | Agent 记忆召回主路径。`text` 是自然语言查询；`layer` 控制 L3/L2/L1；`names` 和 `tagsAny` 是授权后的窄化过滤。 |
| `keyword` | `text`, `limit`, `tagsAny`, `tagsNone` | 直接关键词检索路径，用于测试或内部工具；不做 recall fusion。 |
| `semantic` | `embedding`, `limit`, `tagsAny`, `tagsNone` | 直接向量检索路径；调用者已经提供 embedding。 |
| `entity` | `text`, `entityIds`, `limit`, tag filters | 结构化实体/branch 列表查询。 |
| `relations` | `startEntityId`, `relationTypes`, `maxDepth` | 从实体 id 出发沿 relation graph 扩展。 |
| `workflow` | `text`, `maxDepth`, tag filters | 先按文本找 workflow 相关 entity，再沿 `has`、`depends_on`、`next_is` 扩展。 |

`MemoryRetrievalItem` 字段含义：

| item kind | 字段 | 含义 |
| --- | --- | --- |
| `entity` | `entity`, `branch`, `packedRelations`, `packedBranches`, `evidence`, `score`, `origin` | L3 entity，可附带当前/命中 branch、relation-packed context、证据 chunk、最终分数和来源。 |
| `resource_chunk` | `chunk`, `resource`, `score`, `origin` | L1 原始资源片段候选。 |
| `relation` | `relation`, `depth`, `score`, `origin` | relation graph 候选；`depth` 是从起点扩展的跳数。 |

`MemoryQuerySource` 字段含义：

| 方法 | 含义 |
| --- | --- |
| `keywordSearch` | 用关键词在 entity、branch、resource chunk 中找候选。store-backed 实现先 list vector payload，再对 entity/branch 做 substring match；resource chunk 优先走 BM25 index。 |
| `semanticSearch` | 分别搜索 `memory_entities`、`memory_entity_branches`、`resource_chunks` 三个 vector collection。 |
| `entitySearch` | 列出/过滤 entity 与 branch，并组装成 `EntityRetrievalItem`。 |
| `expandRelations` | 从 `startEntityId` 出发，按 relation type 和 maxDepth 走 source -> target 的 relation graph。 |
| `relationsForObject` | 查任一 source/target id 涉及的 active relation，供 L2 recall packing 使用。 |
| `evidenceFor` | 查 entity -> `refers_to` -> resource_chunk 的证据 chunk。 |

## MemoryRecallImplementation

Recall 是候选生成、授权过滤、relation expansion、reranking、top-P result selection、layer shaping 的组合流程；实现位于 `MemoryRetrievalAdapter.recall`。

步骤：

1. 计算 `layer = query.layer ?? "L3"`、`limit = query.limit ?? 10`、`topP = config.retrieval.recallTopP ?? 0.8`、`candidateLimit = max(limit * 4, 20)`。`limit` 只约束最多返回多少个结果，不再等价于 top-N。
2. 如果 query 带 `tagsAny`，把它并入 `taskScope.allowedTags`，让 vector/list/BM25 查询源在源头过滤；后面仍会再跑 `withinQueryTags`。
3. 运行 `keywordSearch(scopedContext, query.text, candidateLimit)`。每个候选用 `normalizeBm25Score(item.score, query.text)` 产生 BM25 signal。
4. 用 `EntityExtractor.extract(query.text)` 抽取最多 8 个实体/关键词片段。当前默认实现 `HeuristicEntityExtractor` 会取大写开头短语和 BM25 token 中长度至少 4 的 token；接口允许替换成 spaCy 等更强 extractor。
5. 对每个 extracted entity text 生成 embedding，调用 `semanticSearch(scopedContext, embedding, candidateLimit)`；只保留 `score >= 0.5` 的 semantic hit，并把相似度作为 semantic signal。
6. 只有 `layer = "L2"` 时做 relation expansion。对已融合前的每个 hit，收集 hit 相关 object id：entity id、branch id、chunk id、resource id、relation source/target id；调用 `relationsForObject` 找 active relation。
7. 每条 relation 都作为 `RelationRetrievalItem` 加入候选；如果 hit 是 entity 且 relation 满足 packing 规则，还会把 relation 放进该 entity item 的 `packedRelations`，作为独立候选信号。
8. 对同一个 item key，semantic/BM25 取最大值，entityBoost 累加；最后用 `fuseScore` 得到最终 score。同一个 item 被 BM25、semantic、entity/relation signal 多路召回时只算一个 item，signals 融合成一个分数；relation-packed candidate 继续遵守现有独立 reranking candidate 规则。
9. 先做 layer shaping，再对按 score 降序排列的候选执行 top-P result selection：计算所有候选最终 score 的总和 `totalScore`，从最高分开始累加，直到累计分数达到或超过 `totalScore * topP`。若需要 `y` 个候选才能达到该阈值，最终返回前 `min(y, limit)` 个候选。`totalScore <= 0` 时不返回无分候选。
10. `execute` 在 recall 后统一应用 `withinTaskScope`、`withinQueryTags`、`withinQueryNames`，然后为 entity item 加载 `evidenceFor`。如果 recall layer 是 L3，输出会剥掉 branch，只保留 entity 级结果和 evidence；同一 entity 只保留最高分候选，再参与 top-P selection。

BM25 归一化：

```txt
bm25_normalized = 1 / (1 + exp(-steepness * (raw_score - midpoint)))
```

参数随 query token 数调整：

| query token 数 | midpoint | steepness |
| --- | --- | --- |
| `<= 3` | `5.0` | `0.7` |
| `<= 6` | `7.0` | `0.6` |
| `<= 9` | `9.0` | `0.5` |
| `<= 15` | `10.0` | `0.5` |
| `> 15` | `12.0` | `0.5` |

Relation boost：

```txt
memory_count_weight = 1.0 / (1.0 + 0.001 * (same_type_relation_count - 1)^2)
entity_boost = hit_score * relation_boost_weight * memory_count_weight
```

`relation_boost_weight`：

| relation type | weight |
| --- | --- |
| `depends_on` | `1.5` |
| `next_is` | `1.5` |
| `contradicts` | `1.0` |
| `supersedes` | `1.0` |
| `refers_to` | `0.8` |
| `has` | `0.5` |
| `relates_to` | `0.5` |

Fusion：

```txt
raw = semantic + bm25 + entityBoost
final = min(raw / maxExpected, 1.0)
```

`maxExpected` 规则：

| signals | maxExpected |
| --- | --- |
| semantic + BM25 + entityBoost | `3.0` |
| semantic + BM25 | `2.0` |
| semantic + entityBoost | `2.0` |
| semantic only | `1.0` |
| BM25 + entityBoost | `2.0` |
| 其他 | `1.0` |

Top-P result selection：

```txt
top_p = config.retrieval.recallTopP ?? 0.8
threshold = total_score * top_p
return first min(y, limit) candidates where cumulative_score(first y) >= threshold
```

`retrieval.recallTopP` 是运行时配置项，不是 Agent-facing search 参数。默认值是 `0.8`；配置文件可以覆盖，但有效值必须在 `(0, 1]`。`limit` 仍由 Agent 传入，用作安全的结果数量硬上限。

Layer shaping：

| layer | 输出 |
| --- | --- |
| `L3` | 只返回 entity item；同一 entity 只保留最高分。`execute` 最终剥掉 branch 细节。 |
| `L2` | 返回 entity item 和 relation item；entity item 可能带 `packedRelations`。 |
| `L1` | 只返回 resource chunk item。 |

Relation packing 规则：

| relation type | 当前实现 |
| --- | --- |
| `has` | 不 pack，只把 relation 自身作为候选。 |
| `relates_to` | 不 pack，只把 relation 自身作为候选。 |
| `depends_on` | 只有 hit object 是 relation source 时 pack。 |
| `refers_to` | 只有 hit object 是 relation source 时 pack。 |
| `contradicts` | source/target 任一端命中都 pack。 |
| `supersedes` | 只有 hit object 是 relation target 时 pack。 |
| `next_is` | source/target 任一端命中都 pack；当前代码 pack 单跳 relation，不执行多跳链式排序展开。workflow 多跳遍历走 `expandRelations`。 |

授权和过滤：

| 过滤点 | 行为 |
| --- | --- |
| PermissionRouter | 检索 request 必须是 `read`、`search` 或 `traverse_relation` 类 action，且通过 RBAC。 |
| source filter | vector/BM25/list 查询使用 trusted `rootEntityId`、`branchRef`、active status、allowed/denied entity/resource/tag 过滤。 |
| `withinTaskScope` | 二次过滤 entity/resource/relation；relation 还受 `relationExpansionPolicy.allowedRelationTypes` 和 `maxDepth` 限制。 |
| `withinQueryTags` | 对 entity/branch tag 做 `tagsAny`/`tagsNone` 过滤；relation 自身不按 tag 过滤；L1 resource chunk 只在 recall L1 通过。 |
| `withinQueryNames` | `names` 只匹配 entity item 的可见 name/title/description/branch title/branch description/entity id。 |
