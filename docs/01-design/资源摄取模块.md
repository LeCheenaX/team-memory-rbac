# 资源摄取模块

资源摄取模块处理 L1 原始证据：文件、文档、对话 capture、工具输出和 chunk。普通语义记忆写入走[[记忆模块.md#AgentFacingCaptureInput]]；原始文件和长文档必须先进入 Resource/CAS，再由 ingestion 生成 chunks、embedding、BM25 和 evidence links。

代码出处：`src/resources/service.ts`、`src/ingestion/service.ts`、`src/ingestion/bm25.ts`、`src/ingestion/index.ts`、`src/adapters/cas/object-store.ts`、`src/adapters/cas/filesystem.ts`、`src/adapters/libsql/bm25-index.ts`、`src/adapters/libsql/relation-store.ts`、`src/adapters/qdrant/vector-memory-store.ts`。

本模块术语：

| 名称 | 含义 |
| --- | --- |
| L1 原始证据 | 原始文件、文档、对话片段、工具输出等可追溯材料。 |
| Resource/CAS | Resource 是逻辑资源记录；CAS 是按内容哈希存储原始 bytes 的 content-addressed storage。 |
| revision | Resource 某一次内容版本。 |
| chunk | 从 Resource revision 切出的可检索文本片段。 |
| evidence link | L2 语义对象通过 `MemoryRelation type:"refers_to"` 指向 L1 resource/chunk 的证据关系。 |

## Resource

`Resource` 是 L1 原始材料的逻辑对象。

Agent 可负责字段：`title`、`sourceType`、`content` 或文件引用、metadata。

系统负责字段：`id`、`rootEntityId`、`contentHash`、`revisionId`、`createdAt`、`updatedAt`、CAS location、commit metadata。

字段含义：

| 字段 | 含义 |
| --- | --- |
| `title` | 人可读资源标题，不要求全局唯一。 |
| `sourceType` | 资源来源类别，例如 document、conversation、tool_output、code_file。 |
| `content` | 本次导入的文本内容；大文件可由 host 传文件引用而不是让 Agent 粘贴全文。 |
| `metadata` | 描述资源自身的结构化补充信息，例如来源 URL、文件名、mime type。 |
| `id` | 系统生成的 Resource id。 |
| `rootEntityId` | 资源所属记忆 root。 |
| `contentHash` | CAS 内容哈希，用于确认 bytes 可读且未损坏。 |
| `revisionId` | 当前 active revision id。 |
| CAS location | CAS 后端中的物理位置或对象键，不暴露给普通 Agent。 |

示例调用：

```json
{
  "title": "Riverfront release notes",
  "sourceType": "document",
  "content": "Release checklist and evidence text..."
}
```

预期结果：

```json
{
  "resource": {
    "id": "resource:generated",
    "title": "Riverfront release notes",
    "sourceType": "document"
  },
  "revisionId": "revision:generated",
  "contentHash": "sha256:generated"
}
```

## ResourceRevision

每次 Resource create/update 都生成 revision，并写入[[操作记录模块.md#MemoryCommit]] 或 resource-specific commit。

必须字段：`resourceId`、`revisionId`、`contentHash`。

可选字段：`metadata`、`previousRevisionId`。

字段含义：

| 字段 | 含义 |
| --- | --- |
| `resourceId` | revision 所属 Resource。 |
| `revisionId` | 该内容版本的 id。 |
| `contentHash` | 该 revision bytes 的 CAS hash。 |
| `metadata` | 版本级元数据，例如导入来源、解析器版本。 |
| `previousRevisionId` | 前一个 revision id，用于版本链和差异审计。 |

示例结果：

```json
{
  "resourceId": "resource:riverfront-release-notes",
  "revisionId": "revision:2",
  "previousRevisionId": "revision:1",
  "contentHash": "sha256:new"
}
```

预期结果：旧 revision 保留，active resource 指向最新 revision。

## ResourceChunk

`ResourceChunk` 是可检索证据片段。

系统负责字段：`id`、`resourceId`、`revisionId`、`chunkIndex`、`text`、`embedding`、BM25 rows、metadata。

字段含义：

| 字段 | 含义 |
| --- | --- |
| `id` | chunk id，由 ingestion 生成。 |
| `resourceId` | chunk 所属 Resource。 |
| `revisionId` | chunk 来源的 ResourceRevision。 |
| `chunkIndex` | chunk 在 revision 内的顺序号。 |
| `text` | chunk 的文本内容。 |
| `embedding` | 系统计算的向量表示。 |
| BM25 rows | 系统写入的关键词索引行。 |
| `metadata` | chunk 级补充信息，例如 headingPath、filePath、startLine、endLine、tokenCount。 |

示例结果：

```json
{
  "id": "chunk:riverfront-release-notes:0",
  "resourceId": "resource:riverfront-release-notes",
  "revisionId": "revision:2",
  "chunkIndex": 0,
  "text": "OpenClaw pushes support ticket summaries to Riverfront."
}
```

预期结果：搜索 L1 时可以返回 chunk；语义事实升 L2 时用[[记忆模块.md#MemoryRelation]] `type:"refers_to"` 链接 evidence。

## IngestionService

Ingestion 负责 chunking、embedding、BM25 和投影刷新。

必须字段：`resourceId`、`revisionId`。

可选字段：`chunkSize`、`embeddingProvider`、`force`。

字段含义：

| 字段 | 含义 |
| --- | --- |
| `resourceId` | 要摄取的 Resource。 |
| `revisionId` | 要摄取的具体内容版本。 |
| `chunkSize` | chunking 参数；普通 Agent 不需要传。 |
| `embeddingProvider` | 使用哪个 embedding provider；生产配置负责。 |
| `force` | 是否强制重建已有 chunk/index/vector。 |
| `chunksCreated` | 本次生成的 chunk 数量。 |
| `bm25Updated` | BM25 索引是否刷新。 |
| `vectorsUpdated` | vector store 是否刷新。 |

示例调用：

```json
{
  "resourceId": "resource:riverfront-release-notes",
  "revisionId": "revision:2",
  "force": false
}
```

预期结果：

```json
{
  "status": "ingested",
  "chunksCreated": 3,
  "bm25Updated": true,
  "vectorsUpdated": true
}
```

## EvidenceLink

Evidence link 是 `MemoryRelation` 的一种：semantic object `refers_to` L1 resource/chunk。

字段含义：

| 字段 | 含义 |
| --- | --- |
| `subject` | L2 语义对象 endpoint，通常是 `memory_entity_branch`。 |
| `object` | L1 证据 endpoint，通常是 `resource` 或 `resource_chunk`。 |
| `type` | 必须是 `refers_to`。 |
| endpoint `name` | 已导入且授权可见对象的名字或稳定标识；Agent 不得伪造 generated chunk id。 |

示例调用：

```json
{
  "operations": [
    {
      "target": "memory_relation",
      "op": "create",
      "type": "refers_to",
      "subject": {
        "target": "memory_entity_branch",
        "name": "Riverfront receives OpenClaw ticket summaries",
        "parent": "Riverfront"
      },
      "object": {
        "target": "resource_chunk",
        "name": "chunk:riverfront-release-notes:0"
      }
    }
  ]
}
```

预期结果：检索该 branch 时返回 evidence chunk；但 Agent 仍不能直接伪造 generated chunk ids，必须引用已导入/可见资源。

## BM25Index

BM25 index 提供 keyword recall，和 vector recall 一起进入 reranking。

字段含义：

| 字段 | 含义 |
| --- | --- |
| document id | BM25 内部文档键，对应 chunk 或 memory projection。 |
| terms | 从文本中提取的关键词/token。 |
| score | 查询时产生的关键词匹配分数，进入 reranking。 |

示例输入：`ResourceChunk` 或 `MemoryEntityBranch` projection。

预期结果：`MemorySearchInput` 可同时从 BM25、vector、SQL active view 和 relation graph 返回候选。

## VectorMemoryStore

Vector store 保存实体、branch、chunk 的 embedding 和 payload。

字段含义：

| 字段 | 含义 |
| --- | --- |
| embedding | 系统生成的向量。 |
| payload | vector 命中后回查对象所需的最小元数据。 |
| similarity | 查询向量和候选向量的相似度分数。 |

示例输入：投影后的 `MemoryEntityBranch`。

预期结果：相似事实可用于 branch dedupe；检索时参与候选召回。
