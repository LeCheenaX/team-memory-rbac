# 操作记录模块

操作记录模块保存 write path 的事实来源：commit、operation、幂等、冲突、撤销、恢复、重做。Agent 不直接写 History 结构；Agent 只提交[[记忆模块.md#MemoryCaptureOperation]]，系统把它们包装为 `MemoryCommit`。

代码出处：`src/contracts/history.ts`、`src/history/authority.ts`、`src/history/cloud-authority.ts`、`src/history/store.ts`、`src/history/index.ts`、`src/adapters/libsql/history-authority.ts`、`src/adapters/libsql/history-schema.sql`、`src/adapters/legacy-postgres/cloud-memory-authority.ts`。

相关模块：记忆投影见[[记忆模块.md#MemoryProjector]]；同步 pending/resolve 见[[同步模块.md#PendingOperation]]；资源 revision 见[[资源摄取模块.md#ResourceRevision]]。

本模块术语：

| 名称 | 含义 |
| --- | --- |
| History | 系统的追加式事实日志；active view 由它投影出来。 |
| commit | 一次逻辑变更单位，包含一组有序原子 operation。 |
| operation | commit 内的一条原子变更，例如 create entity、create branch、create relation。 |
| idempotency | 防止同一低层请求重复提交造成重复 commit 的机制。 |
| sync/write conflict | 多端或并发写入产生的操作冲突，不等同于记忆语义里的 `contradicts`。 |

## MemoryCommit

`MemoryCommit` 是一次逻辑变更的提交单位。一次 `team_memory_capture` 调用应映射为一个 commit，内部包含一组 ordered atomic operations。

系统负责字段：`id`、`rootEntityId`、`branchRef`、`clientMutationId`、`author`、`createdAt`、`sequence`、`status`。

Agent 负责字段：无。Agent 只负责 capture input。

字段含义：

| 字段 | 含义 |
| --- | --- |
| `id` | commit id，由 History 系统生成。 |
| `rootEntityId` | commit 所属记忆 root。 |
| `branchRef` | commit 作用的逻辑分支。 |
| `clientMutationId` | 幂等键，由系统或 trusted adapter 生成，不由 Agent 提供。 |
| `author` | trusted session 解析出的用户/Agent/host provenance。 |
| `createdAt` | commit 创建时间。 |
| `sequence` | root/branch 内的提交顺序号或等价排序标记。 |
| `status` | commit 状态，例如 pending/applied/reverted/conflicted。 |
| `operations` | 该 commit 包含的有序内部 operation 列表。 |

示例输入见[[记忆模块.md#AgentFacingCaptureInput]]。

预期 commit：

```json
{
  "id": "commit:generated",
  "rootEntityId": "root:riverfront",
  "branchRef": "main",
  "clientMutationId": "generated-by-team-memory",
  "status": "applied",
  "operations": [
    { "kind": "create_entity", "name": "Riverfront" },
    { "kind": "create_branch", "name": "Riverfront naming preference" }
  ]
}
```

验收：如果 Agent payload 里出现 `clientMutationId`，文档测试应判失败；低层 commit 可包含系统生成的 `clientMutationId`。

## MemoryOperation

`MemoryOperation` 是 commit 内的原子变更，来自 Agent-facing `operations[]` 的系统转换结果。

必须字段：`kind`、目标对象必要字段。

系统字段：operation id、resolved ids、timestamps、sequence。

字段含义：

| 字段 | 含义 |
| --- | --- |
| `kind` | 内部原子动作类型，例如 `create_entity`、`create_relation`。 |
| `targetId` | 被修改对象的内部 id；由系统解析。 |
| `sourceId` / `targetId` | relation 两端对象的内部 id；只出现在内部 operation。 |
| `relationType` | 内部 relation type；Agent-facing 输入字段名必须是 `type`。 |
| `sequence` | operation 在 commit 内的顺序。 |

示例转换：

```json
{
  "target": "memory_relation",
  "op": "create",
  "type": "has",
  "subject": { "target": "memory_entity", "name": "Riverfront release checklist" },
  "object": {
    "target": "memory_entity_branch",
    "name": "Confirm latest ticket summaries",
    "parent": "Riverfront release checklist"
  }
}
```

预期内部 operation：

```json
{
  "kind": "create_relation",
  "relationType": "has",
  "sourceKind": "memory_entity",
  "sourceId": "resolved:entity:riverfront-release-checklist",
  "targetKind": "memory_entity_branch",
  "targetId": "resolved:branch:confirm-latest-ticket-summaries"
}
```

验收：`relationType` 可以存在于内部 operation，但 Agent-facing input 必须使用 `type`。

## IdempotencyRecord

幂等记录防止低层重复提交。它属于系统/adapter，不属于 Agent tool schema。

必须字段：`rootEntityId`、`branchRef`、`clientMutationId`、`requestKind`、`requestJson`、`resultJson`。

字段含义：

| 字段 | 含义 |
| --- | --- |
| `rootEntityId` | 幂等记录所属 root。 |
| `branchRef` | 幂等记录所属 branch。 |
| `clientMutationId` | 本次请求的幂等键。 |
| `requestKind` | 请求类别，例如 write/import/push。 |
| `requestJson` | 首次执行时的规范化请求内容。 |
| `resultJson` | 首次执行后的规范化结果，用于重复请求直接返回。 |

示例：

```json
{
  "rootEntityId": "root:riverfront",
  "branchRef": "main",
  "clientMutationId": "generated-by-team-memory",
  "requestKind": "write"
}
```

预期结果：同一个 `clientMutationId` 重放相同 request 返回相同 result；重放不同 request 报错。

## ConflictRecord

冲突记录描述 local/remote 或 concurrent writes 的冲突。语义矛盾本身应通过[[记忆模块.md#MemoryRelation]] 的 `type:"contradicts"` 表达；这不同于 sync/write conflict。

必须字段：`conflictKey`、`localOperation`、`remoteOperation`、`status`。

可选字段：`resolution`、`resolvedBy`、`resolvedAt`。

字段含义：

| 字段 | 含义 |
| --- | --- |
| `conflictKey` | 冲突对象的稳定定位键，由系统生成。 |
| `localOperation` | 本地 pending operation 或其摘要。 |
| `remoteOperation` | Cloud/concurrent operation 或其摘要。 |
| `status` | 冲突状态，例如 open/resolved/ignored。 |
| `resolution` | 采用的解决方式，例如 keep_local、keep_remote、manual。 |
| `resolvedBy` | 执行解决的人或系统主体。 |
| `resolvedAt` | 冲突解决时间。 |

示例结果：

```json
{
  "status": "conflict",
  "conflictKey": "memory_entity_branch:Riverfront:Weekly report audience",
  "localOperation": "update_metadata",
  "remoteOperation": "create_branch"
}
```

预期结果：active view 不自动接受冲突变更；需要 resolver 选择 keep-local、keep-remote 或 manual。

## RestoreAndRedo

恢复/重做基于 commit log，不基于 Agent 重新描述事实。

字段含义：

| 字段 | 含义 |
| --- | --- |
| `operation` | 管理动作，例如 `revert_commit` 或 `reapply_commit`。 |
| `commitId` | 要恢复或重做的原 commit id。 |
| `newCommitId` | 系统生成的新 commit id。 |
| `revertedCommitId` | 被 revert 的原 commit id。 |

恢复示例：

```json
{
  "operation": "revert_commit",
  "commitId": "commit:generated"
}
```

预期结果：

```json
{
  "status": "applied",
  "newCommitId": "commit:generated:revert",
  "revertedCommitId": "commit:generated"
}
```

重做示例：

```json
{
  "operation": "reapply_commit",
  "commitId": "commit:generated"
}
```

预期结果：生成新 commit，重新投影 active view；权限仍需重新验证。

## AuditRecord

Audit record 记录谁在什么权限下触发了管理或写入动作。

必须字段：`id`、`rootEntityId`、`actorUserId`、`action`、`createdAt`。

可选字段：`agentId`、`payload`、`decisionId`。

字段含义：

| 字段 | 含义 |
| --- | --- |
| `id` | audit record id。 |
| `rootEntityId` | 被操作的记忆 root。 |
| `actorUserId` | 触发动作的人类用户 id。 |
| `agentId` | 代表用户执行动作的 Agent id。 |
| `action` | 被审计的动作名。 |
| `createdAt` | 审计记录创建时间。 |
| `payload` | 与审计有关的安全摘要，不应保存完整敏感正文。 |
| `decisionId` | 对应的 RBAC 授权决策 id。 |

示例：

```json
{
  "action": "write_memory",
  "actorUserId": "user:lex",
  "agentId": "agent:hermes",
  "payload": {
    "targets": ["memory_entity", "memory_entity_branch"]
  }
}
```

预期结果：审计记录可解释某次 memory active view 变化来自哪个 trusted session 和权限决策。
